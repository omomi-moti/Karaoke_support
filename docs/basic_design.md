## 1. 機能設計

### 1.1 機能一覧

| ID | 機能名 | 概要 | 優先度 |
|----|--------|------|--------|
| F-001 | 歌唱セッション記録 | 曲選択後、Intent（Shout/Emo/Practice）とスコア（0〜100）を入力して保存 | Must |
| F-002 | インサイト表示（タイムマシン） | 過去1ヶ月の歌唱曲ランキングをローカルから表示 | Must |
| F-003 | インサイト表示（マイアンセム） | Intent別の回数ランキング・点数ランキングをローカルから表示 | Must |
| F-004 | ハイブリッド検索 | ローカル優先＋Spotify検索、Debounce 0.5秒 | Must |
| F-005 | Spotify視聴履歴表示 | 最近再生した曲のキャッシュ一覧 | Should |
| F-006 | 履歴一覧・フィルタリング | 日時順表示、Intentフィルター、Infinite Scroll | Must |
| F-007 | 手動曲名入力 | オフライン時等のエスケープハッチ | Must |
| F-008 | データ更新 | アプリ起動時＋手動リフレッシュ | Must |
| F-009 | Spotify OAuth連携 | 初回オンボーディング、トークン管理 | Must |
| F-010 | オフライン対応 | ローカルデータ前提の動作、エラー時再試行UI | Must |

### 1.2 主要ユースケースフロー

#### UC-001: 歌唱記録（オンライン）

1. ユーザーが選曲画面（インテントタブ or Spotify履歴タブ or 検索）で曲を選択
2. Intent選択画面で Shout / Emo / Practice のいずれかを選択
3. スコア（0〜100）を入力、メモ（任意）を入力
4. 保存ボタンをタップ → ボタン非活性化、ProgressView表示
5. ローカルDBに保存（UUIDで冪等性確保）
6. 成功時: 完了画面 or 履歴へ遷移 / 失敗時: 再試行UI表示

#### UC-002: 歌唱記録（オフライン）

1. ユーザーが「手動で曲名を入力して歌う」を選択
2. オフライン時: 「ネットワークに接続してください」＋接続導線（設定・リトライ）を表示。**ただしブロックはせず、曲名入力・保存は可能**（FR-008: エラーを出さずにローカルのみで保存）
3. オンライン時 or ユーザーが接続後: 曲名を入力して検索 or そのまま曲名で記録
4. Intent・スコアを入力して保存
5. Spotify Track IDが取得できない場合は端末内ローカルDBにのみ保存し、外部送信は行わない（FR-008 / FR-012 準拠）

#### UC-003: インサイト閲覧

1. ユーザーが選曲画面の「インテント」タブを開く
2. 歌唱データ0件: 「まず1曲歌ってみよう！」＋タブB or 検索への導線
3. 歌唱データ1件以上: タイムマシン（過去1ヶ月ランキング）＋マイアンセム（Intent別ランキング）を表示
4. ランキング内の曲をタップ → 歌唱記録フローへ

#### UC-004: ハイブリッド検索

1. ユーザーが検索欄にキーワードを入力
2. 入力と同時にローカルDBからインクリメンタルサーチ（歌った回数順）
3. 入力停止0.5秒後にSpotify API検索、結果をローカル結果の下部に追加
4. 結果0件 or オフライン: 「手動で追加して歌う」ボタン表示
5. 曲を選択 → 歌唱記録フローへ

## 2. 技術構成設計

### 2.1 技術スタック

| 項目 | 選定技術 | 採用理由 |
|------|----------|----------|
| **プラットフォーム** | iPhone 専用（iPad非対応） | 要求仕様で対象デバイスをiPhoneのみと規定。画面設計・テスト範囲を限定し開発効率を確保。 |
| **言語** | Swift 5.9+ | iOSネイティブ開発の標準。SwiftData・SwiftUIとの親和性が高い。 |
| **UI フレームワーク** | SwiftUI | 宣言的UIで状態管理が明確。iOS 17+でSwiftDataと統合しやすい。 |
| **ローカルDB** | SwiftData | iOS 17+のネイティブORM。Core Dataの後継として@Modelでスキーマ定義が簡潔。オフラインファーストの永続化層として最適。 |
| **軽量キャッシュ** | インメモリ | Spotify視聴履歴は流動的で件数も限定的。最長24時間のTTLを持つインメモリキャッシュとして実装し、アプリ再起動で初期化される。SwiftDataより軽量で、起動時の即時表示に適する。Spotify メタデータ永続保存禁止に準拠。 |
| **認証情報保存** | Keychain | OAuthトークンの安全な保存。UserDefaultsは平文で不適切。 |
| **ネットワーク** | URLSession + 自前ラッパー | 軽量でSpotify APIのREST呼び出しに十分。Alamofire等は過剰。 |
| **Spotify連携** | OAuth 2.0 PKCE | Spotify Web APIの推奨方式。ネイティブアプリでAuthorization Code + PKCEが必須。 |
| **最低OS** | iOS 17.0 | SwiftDataの安定利用のため。 |

### 2.2 アーキテクチャ方針

- **Repository パターン**: UIはAPIレスポンスを直接参照せず、ローカルDBの変更を監視する。オフラインファーストとSSOT（Single Source of Truth）を実現。
- **冪等性**: クライアント生成のUUIDをIdempotency Keyとし、二重送信・二重登録を防止。

### 3.3 各画面の役割

- **S-001 オンボーディング**: Spotify連携を促す。スキップ時はローカル機能のみ利用可能とする案内。
- **S-002 選曲・ホーム**: 2タブ（インテント / Spotify視聴履歴）。Empty State時は「まず1曲歌ってみよう！」。
- **S-003 検索**: ローカル優先、歌った回数順。Debounce 0.5秒でSpotify検索。0件時は「手動で追加して歌う」。
- **S-004 Intent選択**: 3モードから1つ選択。選択後はS-005へ。
- **S-005 歌唱記録入力**: スコア（0〜100）、メモ（任意）。保存ボタン二重送信防止。
- **S-006 History**: 日時順、Intentフィルター、Infinite Scroll。タブバーまたはナビから常時アクセス可能。
- **S-007 設定**: リフレッシュ、Spotify連携状態、オフライン時のネットワーク導線。
- **S-008 手動曲名入力**: オフライン時は「ネットワークに接続してください」＋接続導線。

### 3.4 Empty State / エラー状態 一覧

| 画面 | 条件 | 表示内容 | ユーザーへのアクション |

## 4. 権限設計

### 4.1 アプリ権限（ローカルアプリとして）

| 権限 | 用途 | 必須/任意 |
|------|------|------------|
| **ネットワーク** | Spotify API呼び出し、トークン取得 | 必須（オフライン時はローカルのみ動作） |
| **Keychain** | OAuthトークン保存 | 必須 |
| **ローカルストレージ** | SwiftData、UserDefaults（UIフラグ等のみ） | 必須 |
| **バックグラウンド** | 特になし（オフライン同期はフォアグラウンドで実行） | 不要 |

### 4.2 iOS プライバシー権限

- **カメラ・マイク**: 使用しない（録音機能なし）。
- **位置情報**: 使用しない。
- **写真ライブラリ**: 使用しない。
- **追跡**: 使用しない。ATT（App Tracking Transparency）は不要。

### 4.3 Spotify OAuth スコープ（最小権限の原則）

v1 では読み取り専用の以下の **1つのみ** を要求する。追加スコープは将来バージョンで段階的に追加する。

| スコープ | 用途 |
|----------|------|
| `user-read-recently-played` | 最近再生した曲の取得（v1 必須） |
| `user-top-read` | トップトラック等（**将来バージョン**で追加予定、v1 では要求しない） |
| `user-library-read` | ユーザーの保存曲（**将来バージョン**で追加予定、v1 では要求しない） |

## 5. 運用設計

### 5.1 クラッシュ時のログ取得方針

- **OS標準**: クラッシュレポートはXcode Organizer / App Store Connectで取得。
- **構造化ログ**: すべてのSpotify API通信でJSON構造化ログを出力（timestamp, request_id, endpoint, status_code, latency_ms, retry_count, user_id_hash, error_message）。
- **PIIマスキング**: user_idはSHA-256ハッシュ化。トークンはログに含めない。
- **デバッグビルド**: 開発時はコンソールにログ出力。本番ではファイルまたはリモート送信は現状未実装（将来検討）。

### 5.2 データのバックアップ・復元方針

- **バックアップ**: 現状は端末内のみ。iCloud同期は将来検討。
- **復元**: アプリ削除時はデータも消える。ユーザーへの事前案内が必要。
- **エクスポート**: 現状未実装。将来はJSON/CSVエクスポートを検討。

### 5.3 データ更新タイミング

- **アプリ起動時**: ローカルDBを読み込み、ネットワークがあればSpotify APIでキャッシュ更新。
- **手動リフレッシュ**: プルダウンまたは設定画面のリフレッシュボタンで実行。
- **歌唱記録保存時**: 即座にローカルDBに反映。UIはRepository経由で自動更新。
