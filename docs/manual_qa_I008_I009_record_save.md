# 手動QA: 記録の保存（I-008 / I-009 相当）

**目的**: 記録シートで **曲名 →（画面上は）スコア → Intent → メモ → 保存** が一貫して動くかを確認する。  
（実装の並びは `RecordingSheetContentView` に準拠）

## 前提

- アプリをビルドしてシミュレータまたは実機で起動する。
- **選曲**タブを開き、右上 **「記録を追加」** でシートを開く。

---

## 必須（Must）

1. **手動で曲名**を入力する（例: `テスト曲A`）。
2. **スコア**を変更する（Slider、0〜100）。
3. **Intent** を1つ選ぶ（Shout / Emo / Practice）。
4. **メモ**を任意で入力する（空でも可）。
5. **「保存する」**をタップする。

**期待結果**

- インラインエラーが出ない。
- 保存中はオーバーレイ＋`ProgressView` が出て、**保存中はボタンが押せない**（I-010）。
- 成功後、**シートが閉じる**。
- **履歴**タブに自動で切り替わる（`RootView` の `selectedTab = .history`）。
- **履歴画面の一覧表示**（直近の記録が見えること）は **I-014** の範囲。現状 `HistoryRootView` がプレースホルダーの場合は、**タブ切替まで**を必須の合格とし、データ確認は SwiftData デバッグ／I-014 後に行う。

---

## 任意（保存失敗と再試行）

ローカル SwiftData のみの保存では、通常は失敗しにくい。**意図的に失敗を再現する**場合の例:

- **プレビュー / テスト用**: `PreviewSessionRepository` などで `saveNewRecordingSession` が `throw` するように一時変更してビルドする。
- **デバッガ**: `SwiftDataSessionRepository.saveNewRecordingSession` の `try modelContext.save()` の直前にブレークし、例外を投げる／`save` をスキップするなど（上級者向け）。

**再試行時に見たいこと（I-010 / I-011）**

- エラー表示＋**再試行**で、保存処理が走ること。
- **保存中は** `isSaving` により **連打・二重タップで同時に 2 回 `save()` が走らない**こと（UI 層 I-010）。
- **データ層（I-011）**: 同一 **`SingingSession.id`（Idempotency Key）** での再送は、**既に DB にあれば insert ・ `singCount` 加算をしない**（同一キーの再送は無視）。**初回が未保存で失敗した**再試行では、同じキーで初回 insert される想定。

---

## 任意（失敗 → スコア変更 → 再試行）※境界確認

**目的**: V1 では **失敗後の再試行は同じ内容を推奨**（`docs/v1_issues.md` I-011 補足）。**入力を変えたあと再試行**したときの挙動を確認し、**期待とズレる場合は Issue 化**する。

**手順**

1. 上記の方法で **1 回目の保存を失敗**させる（エラー＋再試行表示まで）。
2. **スコアだけ**変える（他はそのままでも可）。
3. **再試行**（または「保存する」）をタップする。

**確認すること**

- **1 回目が DB に保存されていない**（`exists` が false）場合: 変更後の内容で **新規保存**されるか。
- **1 回目が実は保存済み**だった／**同一キーが既に存在**する場合: 冪等により **更新されず成功に見える**可能性がある（境界）。**プロダクトとして許容するか**を判断し、問題なら **Issue に再現手順と期待挙動**を書く。
