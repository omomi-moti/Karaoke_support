# Feature Specification: ヒトカラモバイルiOS

**Feature Branch**: `001-hitora-karaoke-ios`  
**Created**: 2026-03-12  
**Status**: Draft  
**Input**: docs/raw_spec.md v4.1 に基づく要求仕様書（Spotify API規約・メタデータ非永続化対応）

## Assumptions / Dependencies *(mandatory)*

- **プラットフォーム前提**
  - 対応OS: iOS 17 以降（Apple Silicon シミュレータ / 実機の両方で動作検証）
  - ターゲット端末: iPhone（iPad 最適化は本イテレーション対象外）
- **Spotify 連携に関する前提**
  - ユーザーは有効な Spotify アカウント（v1 は **Spotify Free プランを前提**とし、Premium 固有機能は本イテレーション対象外）を所持していること
  - 認可フロー: Spotify OAuth（Authorization Code with PKCE）を使用
  - 必須/候補スコープ（`docs/basic_design.md` の最小権限ポリシーと整合）:
    - `user-read-recently-played`（最近再生トラックの参照。v1 での必須）
    - `user-top-read`（ユーザーのトップトラック/アーティスト参照。将来の推薦・検索強化向け候補）
    - `user-library-read`（ユーザーライブラリからの曲検索・選択を許可する場合の候補）
  - Spotify の利用規約および開発者ポリシーに従い、**楽曲メタデータを永続キャッシュしない**（楽曲ID・参照用キーのみローカル保持可）。
- **データ保持 / オフライン同期の前提**
  - 紐付けの正本キーは `spotifyTrackId` とし、Track エンティティに永続化する（必須）。
  - 歌唱セッション（Intent・スコア・日時などのユーザー入力データ）はローカルDBに永続化してよい。
  - 楽曲メタデータは原則としてオンライン時に Spotify API から都度取得し、最小限の一時キャッシュのみを許容する。
  - 表示用メタデータのキャッシュはインメモリのみとし、最長24時間以内のTTLを持つ。アプリ再起動時に必ず破棄される。
  - キャッシュ破棄後も `spotifyTrackId` を用いてオンライン時に再取得し、表示情報を復元できることを前提とする。
  - オフライン時も歌唱セッションのローカル保存は可能とし、オンライン復帰後も**端末内ローカルデータとしてのみ扱い、外部サーバーへの同期は行わない**。
  - バックエンド/API の有無:
    - v1 は **完全オンデバイス（クライアントローカルのみ）構成**とし、本 spec の範囲ではバックエンド / クラウド同期API は**前提としない**（将来バージョンでの拡張対象とする）。
- **プライバシー / 法務関連の前提**
  - プライバシーポリシーはプロダクト公式サイト配下（例: `https://example.com/privacy/hitora`）にホスティングされる。
  - アプリ内からは設定画面またはオンボーディングで当該URLへのリンクを表示する。
  - 個人情報として扱うデータ（メールアドレス等）は、本イテレーションではアプリ外（例: Apple ID / Spotify 側）で管理され、アプリ側では**保持しない**前提。
- **分析 / ロギングの前提**
  - アナリティクスツール（例: Firebase Analytics 等）の導入有無は別途決定とし、本 spec では**最小限のイベントロギングのみをアプリ内で定義**する。
- **非機能前提**
  - ネットワークタイムアウトや Spotify API の一時的な失敗はユーザーに致命的なエラーとしてではなく、リトライ可能な状態として扱う。
  - 大規模スケール（数十万ユーザー）向けのパフォーマンス最適化は本イテレーションの範囲外。
## User Scenarios & Testing *(mandatory)*

### User Story 1 - 歌唱セッションの記録 (Priority: P1)

ユーザーが歌った曲を、感情・文脈（Intent）とスコアとともに記録し、後から振り返れるようにする。

**Why this priority**: 記録がなければ振り返りもインサイトも成立しない。アプリのコア価値の根幹。

**Independent Test**: 曲を選択し、Intent（Shout/Emo/Practice）とスコア（0〜100）を入力して保存できる。履歴画面で記録が日時順に表示される。

**Acceptance Scenarios**:

1. **Given** 曲選択画面で曲を選んだ状態, **When** Intentとスコアを入力して保存する, **Then** 歌唱セッションが保存され、履歴に表示される
2. **Given** ネットワーク未接続の状態, **When** 手動で曲名を入力して歌うを選択する, **Then** 「ネットワークに接続してください」等の案内を表示しつつも、手動入力による曲名入力・保存はブロックせず継続でき、接続設定画面等への導線が用意されている
3. **Given** オフラインで歌唱を記録した状態, **When** 保存する, **Then** エラーを出さずにローカルに保存され、歌唱データ本体はローカルのみで保持される。オンライン復帰時には Spotify Track ID の補完および必要なメタデータ再取得のみを行う
4. **Given** 同一曲を複数回歌った状態, **When** それぞれ記録する, **Then** 各歌唱セッションとして Intent と点数・日付が個別に追記されつつ、曲情報（Trackメタデータ）は重複保存されない

---

### User Story 2 - インサイトによる選曲支援 (Priority: P2)

過去の歌唱データに基づく「タイムマシン」「マイアンセム」インサイトを表示し、次に歌う曲の選定を支援する。

**Why this priority**: Spotify単体では得られない独自価値。記録の蓄積が選曲の意思決定を支援する。

**Independent Test**: 歌唱データが1件以上ある状態で、インテントタブを開くとタイムマシン（過去1ヶ月ランキング）とマイアンセム（Intent別の回数・点数ランキング）が表示される。

**Acceptance Scenarios**:

1. **Given** 歌唱データが1件以上ある状態, **When** インテントタブを表示する, **Then** タイムマシン（過去1ヶ月の歌唱曲ランキング）が表示される
2. **Given** 歌唱データが1件以上ある状態, **When** インテントタブを表示する, **Then** マイアンセム（Shout/Emo/Practiceごとの回数ランキング・点数ランキング）が表示される
3. **Given** 歌唱データが0件の状態, **When** インテントタブを表示する, **Then** 「まず1曲歌ってみよう！」の導線が表示される

---

### User Story 3 - ハイブリッド検索と選曲 (Priority: P2)

ローカル履歴とSpotify検索を組み合わせ、歌いたい曲を素早く見つけて選べるようにする。

**Why this priority**: 選曲の効率化は頻繁に行う操作。オフラインでもローカル検索で選曲可能であることが重要。

**Independent Test**: 検索欄にキーワードを入力すると、ローカル履歴が優先表示され、続いてSpotify検索結果が表示される。結果が0件の場合は「手動で追加して歌う」が表示される。

**Acceptance Scenarios**:

1. **Given** 検索画面を開いた状態, **When** キーワードを入力する, **Then** ローカルDBにある曲が優先表示され、歌った回数が多い順に並ぶ
2. **Given** ネットワーク接続中, **When** キーワード入力後0.5秒経過する, **Then** Spotify検索結果がローカル結果の下部に追加表示される
3. **Given** 検索結果が0件（またはオフライン）の状態, **When** 検索画面を表示する, **Then** 検索キーワードを引き継いだ「手動で追加して歌う」ボタンが表示される

---

### User Story 4 - Spotify視聴履歴からの選曲 (Priority: P3)

最近Spotifyで聴いた曲を一覧表示し、そこから歌唱記録に繋げられるようにする。

**Why this priority**: 外部からのインスピレーション。Spotify連携が前提だが、オフライン時は**同一アプリ起動中（同一セッション）に取得済みの**キャッシュで表示可能。

**Independent Test**: Spotify視聴履歴タブで、同一アプリ起動中（同一セッション）に過去に取得した「最近再生した曲」が表示される。空の場合は「手動で曲名を入力して歌う」が表示される。

**Acceptance Scenarios**:

1. **Given** 同一アプリ起動中にSpotifyから取得済みの履歴がキャッシュされている状態, **When** Spotify視聴履歴タブを表示する, **Then** キャッシュされた曲一覧が表示される
2. **Given** 同一アプリ起動中のキャッシュが空の状態, **When** Spotify視聴履歴タブを表示する, **Then** 「手動で曲名を入力して歌う」ボタンが表示される

---

### User Story 5 - 履歴の閲覧とフィルタリング (Priority: P3)

記録した歌唱セッションを日時順に一覧表示し、Intentで絞り込めるようにする。

**Why this priority**: 振り返りの基本。大量データ時もパフォーマンスを維持する。

**Independent Test**: History画面で歌唱履歴が日時順に表示される。IntentフィルターでShout/Emo/Practiceに絞り込める。スクロールで追加読み込みされる。

**Acceptance Scenarios**:

1. **Given** 歌唱セッションが1件以上ある状態, **When** History画面を表示する, **Then** 日時順で一覧表示される
2. **Given** History画面を表示した状態, **When** IntentフィルターでShoutを選択する, **Then** Shoutで記録したセッションのみ表示される
3. **Given** 大量の履歴がある状態, **When** スクロールする, **Then** ページネーションで追加読み込みされ、メモリ消費が抑制される

---

### Edge Cases

- **歌唱データ0件時**: インサイトの代わりに「まず1曲歌ってみよう！」を表示。タブBまたは検索への誘導を行う。
- **オフライン時**: その旨を表示し、ローカルデータを前提に動作。手動曲名入力時は「ネットワークに接続してください」＋接続導線を表示。
- **APIエラー・レート制限時**: 再試行を促すUIを表示。指数バックオフで自動リトライする。
- **二重送信**: 保存ボタンタップ後は非活性化し、処理完了まで複数回タップを受け付けない。データ層でも重複登録を防止する。
- **同一曲の複数回記録**: 点数・日付・Intentを各 SingingSession に追記し、曲のメタデータや Spotify Track ID は Track 側で一元管理し重複保存しない。
- **Spotify メタデータの表示**: オフライン時は Track ID のみ保持しているため、メタデータ（曲名等）は表示不可。その場合は「曲情報を取得できません（ネットワーク接続が必要）」等と表示する。手動入力曲名がある場合はそれを表示する。
- **IDのみ保持・メタデータ欠損時**: `spotifyTrackId` は保持しているが曲名等を取得できない場合は、「曲情報を取得できません（オンライン時に再取得）」と明示し、再試行導線を表示する。スコア・Intent・日時・メモなどのローカル情報は通常表示する。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、歌唱セッション（Intent、日時、スコア、メモ、Spotify Track ID）を記録し、ユーザーが削除するまで無期限で保持しなければならない。Spotify API規約のため、曲名・アーティスト名・アートワーク等のメタデータはDBに永続しない。表示時は Track ID を用いて API から動的取得するか、24時間以内のインメモリ一時キャッシュを利用する。
- **FR-002**: システムは、歌唱スコアをユーザーが手動で0〜100の範囲で入力できるようにしなければならない。
- **FR-003**: システムは、3つのIntent（Shout / Emo / Practice）から1つを選択して記録できるようにしなければならない。
- **FR-004**: システムは、インテントタブで「タイムマシン」（過去1ヶ月の歌唱曲ランキング）と「マイアンセム」（Intent別の回数ランキング・点数ランキング）をローカルデータから表示しなければならない。
- **FR-005**: システムは、検索結果をローカルデータ優先で表示し、ローカル内では歌った回数が多い順、その後に外部API結果を表示しなければならない。
- **FR-006**: システムは、曲選択画面で「インテント」タブと「Spotify視聴履歴」タブの2つのみを提供し、歌唱履歴はHistory画面でのみ閲覧可能にしなければならない。
- **FR-007**: システムは、アプリ起動時にデータを更新し、ユーザーが手動でリフレッシュできるようにしなければならない。
- **FR-008**: システムは、ネットワーク接続時は通常どおりローカルへの保存と必要な外部API参照を行い、オフライン時はローカルへの一時保存のみを行うことでエラーを出さずに記録を継続できるようにしなければならない。また、オンライン復帰時にはオフライン中に作成された記録について不足している Spotify 情報の取得・補完のみを試みるものとし、アプリ専用バックエンドなど外部同期先の存在は前提としない。
- **FR-009**: システムは、手動で曲名を入力して歌う際、オフラインの場合は「ネットワークに接続してください」と表示し、接続への導線を用意しなければならない。
- **FR-010**: システムは、同一曲の複数回記録について、Track レコードは再利用し、SingingSession（Intent・スコア・メモ・日時）は都度新規作成する。これにより、曲の基本情報（Track ID など）は重複保存せず、歌唱記録のみを蓄積する形で管理しなければならない。
- **FR-011**: システムは、エラー発生時にユーザーに再試行を促すUIを表示しなければならない。
- **FR-012**: システムは、オフライン時・外部データ取得不可時にその旨を表示し、ローカルデータを前提として極力動作しなければならない。
- **FR-013**: システムは、保存処理の二重送信を防止し（UI層・データ層）、再送時でもデータが二重登録されないようにしなければならない。
- **FR-014**: システムは、日本語のみを対応言語とし、iPhoneのみを対象デバイスとする。
- **FR-015**: システムは、録音機能を提供しない。
- **FR-016**: システムは、Intent・メモ等の機微情報を外部サーバーに送信せず、インサイト生成を端末内で完結させなければならない。
- **FR-017**: システムは、Spotify から取得したメタデータ（曲名、アーティスト名、アートワーク等）を永続保存してはならない。DB に永続化するのは「ユーザーが手動入力したデータ（スコア、Intent、メモ、手動入力曲名）」と「Spotify Track ID」のみとする。
- **FR-018**: システムは、検索結果画面および設定画面に、Spotify のブランドガイドラインに従ったクレジット（例：「Powered by Spotify」ロゴなど）を配置しなければならない。
- **FR-019**: システムは、アプリ内に「データはすべて端末内にのみ保存され、外部送信されない」旨を明記したプライバシーポリシー（Webページ）へのリンクを設置しなければならない。
- **FR-020**: システムは、Spotify メタデータ未取得時（オフライン・TTL切れ・APIエラー）でも `spotifyTrackId` を保持したまま履歴表示を継続し、メタデータ欠損状態をユーザーに明示しなければならない。

### Key Entities

- **歌唱セッション (Singing Session)**: 1回の歌唱記録。永続化するのは Intent（Shout/Emo/Practice）、日時、スコア（0〜100）、メモ、Spotify Track ID（取得可能な場合）のみ。曲名・アーティスト名・アートワーク等のメタデータは永続化しない。同一曲を複数回歌った場合でも毎回新しい Singing Session レコードを作成し、各セッションごとに Intent・日時・スコア等を個別に保存する。一方で、曲情報（Track メタデータ）は重複保存せず、Track エンティティとしてリレーショナルに管理する。
- **楽曲 (Track)**: 曲の識別子。Spotify API の利用規約により、曲名・アーティスト名・アートワーク等の Spotify メタデータの永続保存が禁止されている。そのため、Spotify 由来のメタデータは Track には保持せず、Spotify 側との紐付け用として `spotifyTrackId` のみを保持する（手動入力曲についてはユーザー生成データである `userEnteredName` を永続化可とする）。一方で、再生回数・最終歌唱日時・`createdAt` / `updatedAt` などの集計／監査用フィールドは SwiftData 管理の内部データとして Track に永続化してよい。表示用の Spotify メタデータは API または 24 時間以内のインメモリ一時キャッシュから取得する。
- **インサイト (Insight)**: タイムマシン（過去1ヶ月ランキング）、マイアンセム（Intent別の回数・点数ランキング）。ローカルDBの Track ID と集計情報から生成し、表示用メタデータは API または一時キャッシュから取得する。

## UI Components & Tabs *(Design Details)*

### 曲選択画面のタブ構成

曲選択画面は以下の2つのタブのみを提供する：

#### タブA: 「インテント (Intent)」 - インサイト駆動の選曲

**インサイト種別**:
- **タイムマシン**: 過去1ヶ月で歌った曲のランキングをローカルDBから取得して表示。
- **マイアンセム**: Shout、Emo、Practice ごとの「過去歌った回数ランキング」と「点数ランキング」をローカルDBから取得して表示。
- **初回起動時フォールバック (Empty State)**: 歌唱データが0件の場合、インサイトの代わりに「まず1曲歌ってみよう！」という導線（タブBまたは検索への誘導UI）を表示する。

#### タブB: 「Spotify視聴履歴 (Spotify)」 - 外部からのインスピレーション

- **データソース**: 過去にオンラインで取得した「最近再生した曲」のインメモリキャッシュを優先表示する。キャッシュは最長24時間以内のTTLを持ち、アプリ再起動で初期化される（Spotify メタデータの永続保存禁止に準拠）。
- **Empty State**: キャッシュが空の場合は「手動で曲名を入力して歌う」ボタンを表示する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは、曲を選んでからIntent・スコアを入力し保存するまでを1分以内に完了できる。
- **SC-002**: オフライン状態でも、過去に記録した歌唱履歴の閲覧と、手動入力による新規記録が可能である。
- **SC-003**: 歌唱データが1件以上ある状態で、インテントタブを開いてから3秒以内にインサイトが表示される。
- **SC-004**: 検索キーワード入力後、ローカル検索結果が即座に（1秒以内に）表示される。
- **SC-005**: 大量の履歴（1000件以上）がある状態でも、スクロール時の追加読み込みにより、操作が遅延しない。
- **SC-006**: エラー発生時、ユーザーは再試行ボタンまたは導線から復旧操作を行える。
- **SC-007**: 初回起動から1曲の歌唱記録完了まで、説明書なしで直感的に操作できる。
